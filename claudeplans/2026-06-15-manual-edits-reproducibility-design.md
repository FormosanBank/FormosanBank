# Reproducible manual XML edits — design

**Status:** design (awaiting review)
**Date:** 2026-06-15
**Topic:** Generalize the Nowbucyang-Truku-Thesis `manual_sentences.xml` pattern into a reusable QC capability: a `QC/utilities/` capture script and a `QC/cleaning/` cleaner, working across multi-file corpora.

---

## Problem

Maintainers sometimes need to hand-edit corpus XML — typically in response to `validate_text.py` or `validate_glosses.py` findings that no automated step can fix. Hand edits are fragile:

1. **They are clobbered by reprocessing.** If an earlier pipeline step is re-run (e.g. a corpus is rebuilt from source), the hand edits are lost.
2. **There is no readable record** of what was changed and why.
3. The existing solution (`manual_sentences.xml` in the Nowbucyang dev repo, applied inline by `scripts/pipeline.py`) is **single-XML only** and **hand-written** — the maintainer edits the manual file by hand and reruns the build.

The Nowbucyang precedent ([`scripts/pipeline.py`](../../Formosan-Nowbucyang-Truku-Thesis/scripts/pipeline.py), lines ~2374–2538) stores hand-curated `<S>` blocks in `data/manual/manual_sentences.xml` under a `<MANUAL_SENTENCES>` root, with two semantics keyed on `id`: **override** (a manual `<S>` whose id matches an automated one replaces it) and **add** (a new id is appended). It has no deletion, no changelog, and assumes one XML file.

## Goal

Generalize this into two cooperating scripts plus a data file and a changelog, usable in any `Formosan-<CORPUS>/` dev repo as part of the QC pipeline:

- **`QC/utilities/capture_manual_edits.py`** — the maintainer hand-edits the XML, then runs this to record the edits into the manual file (so a rebuild can reproduce them). Dumb snapshotter.
- **`QC/cleaning/apply_manual_edits.py`** — re-applies the recorded edits as the first step of the cleaning pipeline, prunes entries that have become no-ops, and emits a human-readable changelog. Smart half.

Both operate on **multi-file corpora** and live outside the XML tree so validators/cleaners don't walk them.

---

## Where this sits in the pipeline

`apply_manual_edits.py` runs **first** in the cleaning pipeline, *before* `clean_xml.py` / `standardize.py` / `add_phonology.py`:

```
build (corpus-specific, in dev repo)        → produces XML  ── this is O at apply-time
  └─ apply_manual_edits.py   (NEW, first)   → re-applies recorded manual edits
     └─ clean_xml.py
        └─ standardize.py
           └─ add_phonology.py
              └─ validators (validate_text, validate_glosses, …)
```

Because `apply_manual_edits.py` runs before standardization and phonology, **the manual file never needs a standard-tier FORM or any PHON element** — those tiers are regenerated downstream. The manual file holds only the pre-standardization content: original-tier `FORM`, `TRANSL`, and `W`/`M` structure.

### Why hand edits happen, and the editing workflow

Hand edits are usually triggered by `validate_text` / `validate_glosses` output, which is produced on **standardized, phonologized** XML. So the maintainer's loop is:

1. Run the full pipeline (through standardize + phonology + validators) to get the findings CSVs.
2. Decide what to correct.
3. Re-run the pipeline **up through the first `clean_xml.py`** to get the working XML at the editable stage.
4. Hand-edit the relevant `<S>` blocks.
5. Run `capture_manual_edits.py` to record the edits.
6. Re-run the pipeline; `apply_manual_edits.py` reproduces the edits.

## Data model

### Manual-edits file — `CodeAndDocs/manual_edits.xml`

One file per corpus. Stores **full `<S>` blocks** (not a diff), grouped by source file:

```xml
<MANUAL_EDITS>
  <FILE path="Amis/story01.xml">          <!-- path relative to --corpora_path -->
    <S id="S023"> … verbatim edited S, standard-tier FORM and PHON stripped … </S>
    <S id="S099" action="delete"/>         <!-- remove the S with this id on apply -->
  </FILE>
  <FILE path="Amis/story02.xml"> … </FILE>
</MANUAL_EDITS>
```

- Root `<MANUAL_EDITS>` → `<FILE path="…">` groups → `<S>` blocks.
- `path` is relative to the corpus's XML root (the `--corpora_path` both scripts are given).
- Default `<S>` action is **upsert** (replace-by-id, or append if the id is new). `action="delete"` records a hand-deletion.
- Recorded `<S>` blocks have **standard-tier FORM and all PHON removed** (see `strip()` below).

### Changelog — `CodeAndDocs/manual_edits.md`

Regenerated by `apply_manual_edits.py` on every run. Per-`<S id>` entries (an "item" = a distinct id), grouped by file, each showing the action and a before → after:

```markdown
# Manual edits — Amis/story01.xml

## S023 — changed
- before (build output):  <original-tier rendering of O[S023]>
- after  (manual edit):   <original-tier rendering of R[S023]>

## S099 — deleted
- before (build output):  <original-tier rendering of O[S099]>
```

Regenerating every run is intentional and safe: when nothing changed, git produces no diff.

### Location & CLI

Both scripts default `--manual_file` to `<corpus-root>/CodeAndDocs/manual_edits.xml`, derived as the sibling `CodeAndDocs/` of the XML root; overridable with `--manual_file`. The changelog is the same path with a `.md` suffix (i.e. `CodeAndDocs/manual_edits.md`). `CodeAndDocs/` is the repo's designated home for reproducibility infrastructure and is outside the XML tree, so the QC walkers never see these files.

## `strip()` — the canonical comparison/storage basis

Both scripts reduce an `<S>` to its **manual-relevant content** before comparing or storing: remove S-level `FORM[@kindOf="standard"]` and every `PHON` (at S, W, and M level). What remains: the `id`/`class`/`sclass` attributes, original-tier `FORM`, `TRANSL`, and the `W`/`M` tree (with their own originals/TRANSLs). Two `<S>` are "equal" when their stripped, canonicalized (lxml `etree.canonicalize`) serializations match.

This basis is what makes the two scripts robust to staging differences: the baseline `B` (committed, published XML) carries standard tiers + PHON, while the working tree `W` may be mid-pipeline; comparing only the stripped content neutralizes that.

---

## `capture_manual_edits.py` (QC/utilities) — dumb snapshotter

**Inputs:** `--corpora_path` (the hand-edited working XML root; required), `--manual_file` (default as above), `--baseline-ref` (= **B**, default `HEAD`), `--verbose`.

**References:** **B** = the `<S>` content at `--baseline-ref` via `git show <ref>:<repo-rel-path>`; **W** = the working-tree `<S>`.

**Algorithm** — for each XML file under `--corpora_path` that also exists in B, and for each `<S id>` (union of ids in B and W), comparing `strip(B)` vs `strip(W)`:

| Situation | Action on the manual file |
|---|---|
| `B ≠ W` | record `R := strip(W)` (replaces any prior R for that id) |
| present in B, **absent in W** | record `R := <S id="…" action="delete"/>` |
| `B = W` | leave untouched (additive — prior entries survive) |

That is the whole job. No O, no changelog, no pruning, no `--force`.

**Edge cases**
- A file present in the working tree but **absent from B** is treated as new build output, not hand edits: `capture` **warns to console and skips it** (otherwise it would record the entire file as "edits").
- If `--corpora_path` is not inside a git work tree, or the ref doesn't resolve, `capture` errors out with a clear message (it has no baseline to diff against).
- Re-running with no new edits is a no-op (every `strip(B) = strip(W)`).

## `apply_manual_edits.py` (QC/cleaning) — smart half

**Inputs:** `--corpora_path` (the XML being cleaned — **this is O at runtime**), `--manual_file` (default as above), `--verbose`.

**Reference:** **O** = the current on-disk `<S>` content at the moment this runs (it runs first in the cleaning pipeline, so O is the build output). `apply` is the only script that has O, by construction.

**Algorithm**
- **No manual file** → print `no manual-edits file found at <path>; nothing to do` and exit 0.
- Otherwise, for each `<FILE path>` group and each operation `R`, compare against `O[target]` on the `strip()` basis:
  - **No-op** — an upsert where `strip(R) = strip(O[target])`, or a delete whose target is already absent → **delete `R` from the manual file**, apply nothing, **and warn to console** (e.g. `pruned no-op manual edit: <file> / <id>`). Expected to be rare.
  - **Otherwise** → apply the operation:
    - upsert: replace the `<S>` with matching id, or append to the `TEXT` if the id is new;
    - delete: remove the `<S>` with matching id.
    Record a changelog entry (before = `O[target]`, after = `R`, or "deleted").
- Write back the (possibly pruned) `manual_edits.xml`, and (re)generate `manual_edits.md`.
- Print a per-file summary (applied / pruned counts).

**Why pruning lives here, not in capture:** only `apply` sees O. If the maintainer edits a block back to exactly what the build produces, `capture` faithfully records it (it only knows B), then the next `apply` sees `strip(R) = strip(O)`, prunes the entry, warns, and notes nothing changed. Undo is therefore automatic — no hard-exit / `--force` needed.

---

## Net behavior & the one caveat

- **Add / change / delete** are all captured and applied.
- **Undo is automatic and self-healing.** Editing a block back to the build output → captured → pruned on next apply (with a console warning).
- **Caveat (documented, not enforced):** run `apply` after regenerating and before `capture`. If you `capture` against a regenerated-but-unapplied tree, you record pre-manual content; the next `apply` prunes it as a no-op, so it is self-healing rather than destructive — but the console prune-warning is the signal that this happened.

## Testing

New tests under `tests/` (matching the existing `tests/cleaners/` layout), using `lxml` + `pytest` + `subprocess`, on synthetic multi-file fixtures:

**`capture_manual_edits.py`**
- change: an edited `<S>` is recorded as `strip(W)` under the right `<FILE path>`.
- add: a brand-new id in an existing file is recorded.
- delete: an id present in B but gone from W is recorded as `action="delete"`.
- additive: a pre-existing unrelated entry survives a capture that touches a different id.
- strip: standard-tier FORM and PHON are absent from the recorded block.
- new-file: a file absent from B is warned-and-skipped, not captured wholesale.
- no-git / bad ref: clean error.

**`apply_manual_edits.py`**
- upsert replace, upsert append, delete.
- multi-file: operations routed to the correct files.
- no-op prune: an entry equal to O is removed, a warning is printed, and the changelog reflects it.
- empty/missing manual file: prints the "nothing to do" message, exits 0, touches nothing.
- changelog content: per-id before/after entries grouped by file.
- idempotence: a second apply with no source change leaves both XML and manual file byte-identical.
