**Design docs (sub-project A produced these during test-writing; they are B's primary specs):**

**Canonical reference.** The canonical XML spec now lives in the gitbook at [`en-us/the-bank-architecture/formosanbank-xml-format.md`](../../FormosanBankGitbook/en-us/the-bank-architecture/formosanbank-xml-format.md). The repo's DTD at [QC/validation/xml_template.dtd](QC/validation/xml_template.dtd) and this prose spec have **already drifted**. Examples found in a quick read of the spec:
- Spec describes a `<PHON>` element (IPA transliterations, with `kindOf` semantics paralleling `<FORM>`); DTD has no `PHON` at all.
- DTD allows `class`/`sclass` on `<W>` and `<M>` and `ver` on `<TRANSL>`; spec doesn't document these (the spec has a literal "didn't include..." note acknowledging the gap).
- Spec mandates `<AUDIO>` `start` and `end` "must always be set"; DTD declares them `IMPLIED` (optional).
- Spec defines multi-citation delimiters (`|` for `citation`, `,` for `BibTeX_citation`) and the `audio="segmented"` convention; DTD enforces neither.

Reconciling these is part of B. Decision needed per disagreement: which is authoritative, and update the other.

**Downstream consumers.** FormosanBank XML is consumed by at least one external tool: the [Formosan-MT-Toolkit](https://github.com/FormosanBank/Formosan-MT-Toolkit) (documented in the gitbook on `origin/en-us` only — PR #20 authored by Hunter S on 2026-05-17, not yet merged to `main`). Its pipeline (`fetch_xml.py → clean_xml.py → make_corpus.py → filter_split_corpus.py`) depends on specific XML invariants — most notably that `<TRANSL>` elements carry well-formed `xml:lang` and `kindOf` attributes that distinguish translation source/type for parallel-pair extraction. **B's invariant tests should explicitly cover what downstream consumers need, not only what the prose spec mandates in the abstract.**

The MT toolkit also has its own `clean_xml.py` (same filename as `QC/cleaning/clean_xml.py` here). The doc says it "runs the standard XML cleaning utilities" but it's ambiguous whether it imports FormosanBank's version or has a copy. This is a duplicate/drift hazard worth reconciling — added to Phase B1 below.

A second locally-present repo, `FormosanChineseMT/`, may be the same project at an earlier stage or a separate predecessor — unclear without further investigation. Worth resolving before B touches anything MT-toolkit-shaped.

**Current state.**
- 7 validation scripts in [QC/validation/](QC/validation/) — all CLIs that emit logs and **never `sys.exit(1)`**. Purely informational.
- Both existing CI workflows ([corpus-metrics.yaml](.github/workflows/corpus-metrics.yaml), [token-comparison.yaml](.github/workflows/token-comparison.yaml)) are also purely informational.
- DTD validation is never run in CI; broken XML can land on `main`.
- Many invariants from the prose spec are not asserted in code anywhere (audio `start`/`end` always set; `audio="segmented"` convention; W-level `standard` retains morpheme segmentation; multi-citation delimiters; infix `-x-` formatting; clitic `=` convention; etc.).

### B9.10 — Translation quality check (BLEU, frequent-word) (deferred indefinitely)

**Status (2026-05-31): [DEFERRED INDEFINITELY].** Per user direction (2026-05-31): we don't have good MT at the moment anyway. Listed here to give the OldQCPlan flowchart's "automatic translation check: frequent-word check, BLEU score of MT (if any)" box an explicit home so it isn't re-proposed.

**Sketch (for future revival).** Treat TRANSL elements as the reference; run a MT model over the FORM tier and score against the TRANSL with BLEU/chrF, surfacing low-score S elements for expert review. Frequent-word check (cross-corpus word-frequency comparison) is a cheaper sibling that doesn't need MT and could land first if revived.

**Why not now.** Off-the-shelf MT for Formosan languages produces output too noisy to make BLEU a useful QC signal; would generate too many false-positive findings to action. Revisit when a Formosan-tuned MT model becomes available (likely co-evolves with the ASR work flagged in B9.2's "future work").
