# GitBook stats: single source of truth + port-time population

**Date:** 2026-06-15
**Status:** Design — approved for spec, pre-implementation
**Branch:** `feature/gitbook-sync-part-d` (FormosanBank) + `feature/corpus-page-tooling` (FormosanBankGitbook)
**Builds on:** [2026-06-14-gitbook-sync-part-d-design.md](2026-06-14-gitbook-sync-part-d-design.md) (the corpus-page tooling this extends)

## Purpose

Two coupled fixes to how corpus statistics reach the GitBook:

1. **One source of truth.** Today the per-corpus stats CSVs exist in **two** places: `FormosanBank/statistics/` (the canonical home, regenerated by the `corpus-metrics` CI on every push to main) and `FormosanBankGitbook/statistics/` (a frozen duplicate dropped 2026-04-07 and stale ever since). Both GitBook-side scripts read the stale duplicate. Eliminate the duplicate: point the GitBook readers at the canonical FormosanBank dir and delete the GitBook copy.
2. **Populated-on-port.** Today a freshly-ported corpus's GitBook page ships with a blank stats table until a separate manual sync runs. Make `port-corpus-in` generate the new corpus's stats CSV and inject its tables as part of the port, so the page ships populated.

This collapses Roadmap Part D / Layer 1 "cross-repo stats-CSV regeneration" from a build-a-pipeline item into a path change plus a port-skill step.

## Background (current state, verified 2026-06-15)

- `FormosanBank/statistics/` holds 20 `<Corpus>_corpora_stats.csv` files — the source of truth. `QC/utilities/get_corpus_stats.py` writes them: `--all` for the whole collection (CI path) **or a single corpus by positional arg** (`get_corpus_stats.py Corpora/<Name>` → writes `<repo_root>/statistics/<Name>_corpora_stats.csv`). It reads audio *seconds* from `statistics/audio_durations.csv` (the audio source of truth; never written by CI).
- `FormosanBankGitbook/statistics/` holds a frozen 2026-04-07 duplicate of those 20 CSVs.
- `FormosanBankGitbook/update_corpus_stats.py` reads `STATS_DIR = REPO_ROOT / 'statistics'` (line 21 — the GitBook's stale copy) and injects per-corpus tables into each corpus page plus the `by language` / `by dialect` aggregate tables into `corpora/README.md`. **GitBook never reads the CSVs at publish time** — it renders the committed `.md`; the CSVs are build-time inputs to this injector.
- `FormosanBankGitbook/manage_corpus_pages.py` `check` reads `statistics = root / "statistics"` (line 222 — same stale copy) for its informational "missing stats CSV" probe.

So both GitBook readers point at the stale local copy; the canonical CSVs in `FormosanBank/statistics/` are never consulted by the GitBook.

## Change 1 — Single source of truth

**Nothing about where CSVs are stored changes.** `FormosanBank/statistics/` remains the only place CSVs are written and the source of truth. Only the GitBook-side *readers* are redirected, and the stale duplicate is removed.

1. **`update_corpus_stats.py` stats-dir resolution.** Replace the hardcoded `STATS_DIR = REPO_ROOT / 'statistics'` with a resolver, precedence:
   1. `--stats-dir <path>` CLI flag,
   2. `FORMOSANBANK_STATS_DIR` environment variable,
   3. default `REPO_ROOT / '..' / 'FormosanBank' / 'statistics'` (the sibling checkout).

   If the resolved dir does not exist, **exit with a hard error** naming the fix ("clone FormosanBank as a sibling of FormosanBankGitbook, or pass `--stats-dir <path>`"). Never silently fall back to a local dir (there won't be one after step 3).

2. **`manage_corpus_pages.py check` stats probe.** Derive the stats dir from the already-required `--corpora-path` (which points at `FormosanBank/Corpora`): `stats_dir = Path(corpora_path).parent / 'statistics'`. The `stats_csv_issues` probe reads there. No new flag. (This keeps the lint's "missing stats CSV" check meaningful against the canonical CSVs.)

3. **Delete `FormosanBankGitbook/statistics/`.** First grep-confirm `update_corpus_stats.py` is its only reader. (Audio durations live in `FormosanBank/statistics/audio_durations.csv`; nothing in the GitBook repo references the deleted dir afterward.)

**Expected one-time side effect.** The first `update_corpus_stats.py` run after this reads *current* FormosanBank stats (≈2 months newer than the April copy) and regenerates **every** corpus page's table plus the `corpora/README.md` aggregate — a large but correct diff. Commit it on its own ("refresh all GitBook stats tables from canonical FormosanBank/statistics") so it's reviewable in isolation.

## Change 2 — Port-time stats population (`port-corpus-in` Phase 5)

After the corpus is in `Corpora/<new>` (existing Phase 3) and as part of the GitBook phase, in this order:

1. **(Audio gate — Change 3, runs first if the corpus has audio.)**
2. **Generate the per-corpus CSV:** `get_corpus_stats.py Corpora/<new>` → writes `FormosanBank/statistics/<new>_corpora_stats.csv`. Commit it alongside the corpus (it lands with the corpus PR; CI later regenerates the identical CSV via `--all`, so no conflict — running the canonical generator is not "hand-editing").
3. **Inject the tables:** run `update_corpus_stats.py` in the GitBook repo (sister-read default, or `--stats-dir <FormosanBank>/statistics`). This fills the new page's stats block **and** refreshes the `corpora/README.md` aggregate. Commit in the GitBook repo.
4. **Verify:** `manage_corpus_pages.py check --strict` passes and the new corpus's "missing stats CSV" informational flag is gone.

The page now ships populated. (Note: step 3 also refreshes other pages if FormosanBank/statistics has drifted — in the normal flow Change 1's one-time refresh has already absorbed that, so a port-time run touches mainly the new corpus.)

## Change 3 — Conditional audio-durations step (detect + guide)

In Phase 5, before the `get_corpus_stats` step:

- **Detect:** does the ported XML contain any `<AUDIO>` elements?
- **No audio → skip silently** and proceed to `get_corpus_stats`.
- **Audio present → pause and guide the maintainer** (do not blindly automate — the audio's physical location varies and the tools differ):
  - **Audio available locally** (downloaded into the FormosanBank corpus dir): `update_audio_stats.py Corpora/<new>` — recomputes durations from local audio into `FormosanBank/statistics/audio_durations.csv`.
  - **Audio only on Hugging Face** (`download_audio_data.sh` ported + uploaded): `refresh-audio-stats <new>` — pulls from HF, computes, deletes.
  - Then continue to `get_corpus_stats` (which now reads the populated seconds).
  - **If the maintainer skips:** proceed; the page's audio columns ship as 0/pending (unchanged from today's behavior). Document this in the Phase 6 summary as a known follow-up.

**Ordering is load-bearing:** audio-durations → `get_corpus_stats` → `update_corpus_stats`. `get_corpus_stats` reads `audio_durations.csv`; if the audio step hasn't run, audio seconds come out 0.

This is a documented, guarded conditional, not full automation — chosen because no audio corpus is available to test a fully-automated path, and the "where is the audio" resolution is genuinely a maintainer judgment.

## Out of scope

- No auto-refresh CI (cron / repository_dispatch) for ongoing drift — the port-time run covers new corpora; ongoing drift across the whole collection stays a manual `update_corpus_stats.py` run for now (option (b) from the discussion deliberately deferred — it needs cross-repo write credentials).
- No change to how/where CSVs are generated or stored (`FormosanBank/statistics/`, `get_corpus_stats.py`, the `corpus-metrics` CI are all untouched).
- No full automation of the audio-durations step (detect + guide only).
