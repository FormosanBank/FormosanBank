## Per-element-type breakdown

| Element type | Bad codes |
|---|---|
| `TRANSL` | `en` (470,141), `zh` (445,328), `zh-Hans` (5,860), `zh-Hant` (5,860) |
| `TEXT` | `amis` (1) |

The pattern is clean: **the bad xml:lang values are almost entirely on `TRANSL` elements** (translations into English or Chinese). Source-tier `xml:lang` on `<TEXT>` is correct in all but one file (Whitehorn_Collection's stray `amis`).

This makes sense: corpus authors who hand-wrote TRANSL elements naturally reached for `en` and `zh` (the codes they know from everyday use) rather than the ISO 639-3 forms `eng` and `zho`.

## Recommended remediation strategy

Since the mapping is unambiguous (one-to-one for all 5 bad codes), this is a mechanical bulk rewrite:

1. **Write a one-off script** (`QC/utilities/remediate_iso_639_3.py` or similar) that walks specified corpora, rewrites the four IETF/639-1 forms to their 639-3 equivalents, and writes the changed files back. Preserves all other content.
2. **Run it per-corpus**, committing each corpus's remediation as a separate commit so the diff is reviewable. Order by impact: ePark first (502K attrs), then NTUFormosanCorpus, then the rest.
3. **The `amis` → `ami` case** is one attribute on one element in one file — fix it manually as part of the corresponding corpus commit (Whitehorn_Collection).
4. **Verify after each commit** that `tests/validators/test_validate_xml.py::test_V035_*` still passes and the corpus_metrics token counts haven't shifted unexpectedly.
5. **Once all 14 corpora are remediated**, V035 can be confidently enforced as HARD in CI without breaking existing files.

## Estimated effort

- Script: ~30 minutes (the logic is trivial; the test coverage is what takes time).
- Per-corpus run + commit: ~5 minutes each = ~70 minutes for 14 corpora.
- Total: ~2 hours of focused work, all of which is `Corpora/`-touching (deliberately deferred per the "don't touch Corpora/" directive from the validator refactor session).

## Open questions

- **`zh-Hans` vs `zh-Hant` in Glosbe.** Both files in Glosbe carry the script subtag (Simplified vs Traditional). ISO 639-3 has no way to distinguish them — both collapse to `zho`. Is the script distinction load-bearing for any downstream consumer? If yes, we need to either (a) preserve the original `zh-Hans`/`zh-Hant` value in a different attribute (e.g., a custom `script="Hans"`/`script="Hant"`), or (b) leave Glosbe as-is and accept it as a known V035 exception.
- **The `amis` stray attribute in Whitehorn_Collection.** One element with `xml:lang="amis"` — almost certainly a typo. Spot-check the file to confirm the intended code is `ami` (Amis) and not, say, a TRANSL into a different Amis-shaped language.
- **Order of operations vs B5 cleaner remediation (roadmap item 17).** The cleaner-extension work (B5) also has a Corpora/ remediation step. Coordinate so both remediations happen close together (one branch covering both), or accept that there will be two sequential remediation rounds touching the same files.
