# FormosanBank near-term roadmap

**Date:** 2026-05-27
**Status:** Draft — top-level sketch. Each sub-project still needs its own design + plan before implementation.

## Purpose

Sketch-depth plan covering five near-term initiatives. Each section names a concrete goal, the relevant current-state facts (from investigation), a recommended approach at sketch depth, dependencies on other sub-projects, rough sizing, and the open questions that need answering before drilling in.

The five sub-projects are independent enough that each deserves its own dedicated design pass. This document is the map, not the territory.

## A. Python test infrastructure

**Goal.** Establish a real test runner, conftest, and the first round of unit tests for the highest-risk QC code, with discovery wired into CI so failures block merge.

**Current state.**
- Zero test infrastructure: no `pytest.ini`, no `pyproject.toml`, no `conftest.py`, no `tests/` tree.
- One genuine unit test exists ([QC/test_find_duplicate_sentences.py](QC/test_find_duplicate_sentences.py)) but uses a hand-rolled PASS/FAIL counter, not pytest, and isn't auto-discovered.
- Two corpus-specific scripts named `test_*.py` under [Corpora/YeddaPalemeqBlog/CodeAndDocs/Scripts/](Corpora/YeddaPalemeqBlog/CodeAndDocs/Scripts/) are baseline analyzers, not unit tests — they should probably be renamed to avoid confusion with real tests.
- `requirements.txt` has no test deps (no pytest, hypothesis, coverage).

**Working assumption.** A non-trivial part of this sub-project IS figuring out what good tests look like for this codebase — we won't know the full set up front. The infrastructure should make it easy to *grow* the suite as we discover what to test, not just stand up a one-shot framework.

**Sketch approach.**
1. Add `pytest` + `pytest-cov` (and possibly `hypothesis`) to `requirements.txt` or a separate `requirements-dev.txt`.
2. Create top-level `tests/` directory with `conftest.py` providing fixtures: a minimal valid `<TEXT>` XML, a temp-corpus directory factory, a path to `QC/validation/xml_template.dtd`.
3. Establish a `tests/fixtures/` directory of **intentional-problem XML files** (negative test cases): e.g., XML missing `kindOf="standard"`, with an invalid `xml:lang`, with W/M count mismatches, with audio `start >= end`, etc. Tests assert the validators catch these. This grows over time as we encounter real problems.
4. Migrate `QC/test_find_duplicate_sentences.py` to pytest style as a worked example.
5. Add a third GitHub Actions workflow `tests.yaml` that runs `pytest` on every PR and push. Fail the build on test failure.
6. First targeted test additions, ranked by risk: `QC/utilities/standardize.py` (mutates XML), `QC/cleaning/clean_xml.py` and `clean_nonlatin.py` (mutate XML in place — the most dangerous code in the repo), `QC/count_tokens.py` and `QC/corpus_metrics.py` (drive CI artifacts).
7. Establish a written "saw a problem → wrote a regression test" workflow: minimal steps for any contributor (or me) to add a fixture + test when something breaks. The point is to make the discipline easy enough that it actually happens.

**Standing instruction (continuous).** Treat test-suite growth as an ongoing activity, not a one-shot setup. I should proactively suggest additional tests over time — especially when reviewing diffs, when running into recurring problem patterns, when discovering invariants in B's discovery work, and when reviewing Hunter S's backlog. Each surfacing of a class of problem is a candidate for a new fixture + test.

**Claude Code support.** A `skill` for "add a test for QC script X" that knows the project's fixture conventions. A `skill` for "turn this bug into a regression test" that captures the discipline from item 7. The `pyright-lsp` plugin (already installed, verified working) gives type-checking for free during test writing.

**Dependencies.** None for the infrastructure itself. B's discovery work (Phase B1 below) feeds A with test ideas.

**Rough sizing.** 1 design session, 1 implementation session for infrastructure + first ~5 tests. Subsequent tests are incremental and continuous.

**Open questions for detailed design.**
- Do you want a separate `requirements-dev.txt` or just add to `requirements.txt`?
- Test layout: `tests/test_<module>.py` mirroring `QC/<module>.py`, or by area (`tests/test_validation/`, `tests/test_cleaning/`)?
- Should the existing `QC/test_find_duplicate_sentences.py` be moved to `tests/` (cleaner) or stay where it is (touches less)?
- Coverage threshold to enforce on CI, or just report?
- For the fixtures: store them as standalone XML files in `tests/fixtures/`, or generate them inline in tests via Python helpers? (Standalone is easier to inspect; inline is easier to diff against the test that uses it.)
- For the "saw a problem → regression test" workflow: should the fixture+test ideally land in the same PR that fixes the bug, or in a follow-up PR? (Same-PR is best for discipline; follow-up is more flexible.)

**Post-A review follow-ups (forward-looking, low priority).** From the 2026-05-30 review of sub-project A:
- **Add a Python version matrix to `tests.yaml`** if the project ever supports >1 Python version simultaneously (today: 3.13 only, sufficient).
- **Empirically document the C012 lookup data in test docstrings.** The C012 tests' "Bunun and Thao preserve, others strip" expectation depends on runtime lookup of `Orthographies/Ortho113/<Language>.tsv`. A B-implementer reading just the test file should be able to re-verify the language list without reading the design doc. Verified empirically (2026-05-30): `-` appears as a letter row in `Bunun.tsv` and `Thao.tsv` only.
- **Add a "what xfail means here" note to `test_V081_*`.** V081 is the only xfail in the suite that requires walking `Corpora/` rather than adding a single function. Worth flagging in the test docstring so a B-implementer doesn't underestimate scope.
- **Long-term: an `xfail_b_phase` decorator** that wraps the xfail marker and asserts the reason string references a tracked B roadmap item. Makes "what B owes" greppable. Defer until xfail count grows or B work begins.

## B. XML data tests as CI gates

**Goal.** Turn the existing manual `validate_*.py` scripts into PR-blocking checks, plus add invariant assertions the current scripts don't cover, plus extend `clean_xml.py` with language-aware behavior. Crucially: *figure out what the right invariants are* by reading the canonical spec, comparing existing corpora, and reading the reproducibility code.

**Design docs (sub-project A produced these during test-writing; they are B's primary specs):**
- [2026-05-29-xml-validation-design.md](2026-05-29-xml-validation-design.md) — every check the new `validate_xml.py` should perform (V001–V083). The xfail tests in `tests/validators/test_validate_xml.py` target the rules not yet enforced; B's validator work flips them to passing.
- [2026-05-29-clean-xml-extension-tests-design.md](2026-05-29-clean-xml-extension-tests-design.md) — the language-aware cleaning behaviors `clean_xml.py` should grow (C001–C026). The "Deferred to sub-project B" section enumerates the six pieces of new cleaner infrastructure needed (language-awareness, warning+CSV output, canonical-orthography lookup, `--hard-remove-segmentation` flag, transformation counter output, corpus remediation).

**Canonical reference.** The canonical XML spec now lives in the gitbook at [`en-us/the-bank-architecture/formosanbank-xml-format.md`](../../FormosanBankGitbook/en-us/the-bank-architecture/formosanbank-xml-format.md). The repo's DTD at [QC/validation/xml_template.dtd](QC/validation/xml_template.dtd) and this prose spec have **already drifted**. Examples found in a quick read of the spec:
- Spec describes a `<PHON>` element (IPA transliterations, with `kindOf` semantics paralleling `<FORM>`); DTD has no `PHON` at all.
- DTD allows `class`/`sclass` on `<W>` and `<M>` and `ver` on `<TRANSL>`; spec doesn't document these (the spec has a literal "didn't include..." note acknowledging the gap).
- Spec mandates `<AUDIO>` `start` and `end` "must always be set"; DTD declares them `IMPLIED` (optional).
- Spec defines multi-citation delimiters (`|` for `citation`, `,` for `BibTeX_citation`) and the `audio="segmented"` convention; DTD enforces neither.

Reconciling these is part of B. Decision needed per disagreement: which is authoritative, and update the other.

**Downstream consumers.** FormosanBank XML is consumed by at least one external tool: the [Formosan-MT-Toolkit](https://github.com/FormosanBank/Formosan-MT-Toolkit) (documented in the gitbook on `origin/en-us` only — PR #20 authored by Hunter S on 2026-05-17, not yet merged to `main`). Its pipeline (`fetch_xml.py → clean_xml.py → make_corpus.py → filter_split_corpus.py`) depends on specific XML invariants — most notably that `<TRANSL>` elements carry well-formed `xml:lang` and `kindOf` attributes that distinguish translation source/type for parallel-pair extraction. **B's invariant tests should explicitly cover what downstream consumers need, not only what the prose spec mandates in the abstract.**

The MT toolkit also has its own `clean_xml.py` (same filename as `QC/cleaning/clean_xml.py` here). The doc says it "runs the standard XML cleaning utilities" but it's ambiguous whether it imports FormosanBank's version or has a copy. This is a duplicate/drift hazard worth reconciling — added to Phase B1 below.

A second locally-present repo, `FormosanChineseMT/`, may be the same project at an earlier stage or a separate predecessor — unclear without further investigation. Worth resolving before B touches anything MT-toolkit-shaped.

**Current state.**
- 7 validation scripts in [QC/validation/](QC/validation/) — all CLIs that emit logs and **never `sys.exit(1)`**. Purely informational.
- Both existing CI workflows ([corpus-metrics.yaml](.github/workflows/corpus-metrics.yaml), [token-comparison.yaml](.github/workflows/token-comparison.yaml)) are also purely informational.
- DTD validation is never run in CI; broken XML can land on `main`.
- Many invariants from the prose spec are not asserted in code anywhere (audio `start`/`end` always set; `audio="segmented"` convention; W-level `standard` retains morpheme segmentation; multi-citation delimiters; infix `-x-` formatting; clitic `=` convention; etc.).

**Sketch approach.** Four phases, B1 (discovery) can start before A is done.

*Phase B1 — Discovery: figure out what to test.*
1. Read the canonical XML spec end-to-end and inventory every assertable rule, with a confidence rating per rule (already enforced by DTD / mechanically enforceable / requires interpretation).
2. **Compare `original` vs `standard` tiers across published corpora** to discover what transformations have actually been applied. These are candidates for tests like "standard should differ from original only in punctuation, orthography, and (at S-level) loss of morpheme segmentation." **Critical caveat:** `standard` itself may contain errors. Some discovered "transformations" will be bugs that need to be fixed in the published corpora, not enshrined as expected behavior. The discovery flow is: surface pattern → review with you → decide (bug to fix vs invariant to enforce) → test.
3. **Read each `CodeAndDocs/README.md` and the reproducibility scripts** they describe, across published corpora. This surfaces (a) transformations that have been applied historically and should be tested, and (b) transformations that *should* have been applied but weren't — a different class of bug.
4. Reconcile the spec/DTD drift identified above. Decide which is canonical per disagreement and update the other.
5. Investigate the MT toolkit's `clean_xml.py` (and any other places downstream tooling carries its own copy of FormosanBank cleaners). Decide whether to consolidate to a single source of truth, document an intentional fork, or update one to match the other. Resolve the `Formosan-MT-Toolkit` vs `FormosanChineseMT` identity question as part of this.
6. **ISO 639-3 remediation across published corpora.** Per the XML validation design ([2026-05-29-xml-validation-design.md](2026-05-29-xml-validation-design.md), rule V035), `xml:lang` attributes must use ISO 639-3 codes. A large number of files likely use ISO 639-1 codes instead (most commonly `en` for English rather than `eng`, but possibly others as well). Discovery: grep `xml:lang="…"` across `Corpora/**` and cross-reference against `QC/validation/iso-639-3.txt` to enumerate offenders. Remediation: bulk-rewrite per language family with a one-off script. Anticipate this as a meaningful chunk of corpus-touching work that needs to land before V035 can go HARD in CI without breaking existing files.

*Phase B2 — Hard-gating infrastructure.*
5. Add an `--exit-nonzero-on-findings` (or similar) flag to each `validate_*.py`. Preserve current default behavior so existing manual runs aren't disrupted.
6. New GitHub Actions workflow `xml-validation.yaml` runs the hard gates on every PR touching `Corpora/**`:
   - `validate_xml.py` (DTD) — always
   - `validate_punct.py` — always
   - `validate_glosses.py` — only for corpora with `W`/`M` segmentation
7. A second tier of "soft" checks (orthography, vocabulary) that comment on PRs but don't block — see B4 for thresholds.

*Phase B3 — New invariant tests.*
8. New invariant tests under `tests/test_invariants/`: one test per assertable rule from B1, including the ones the prose spec mandates but the DTD doesn't capture (audio start/end always set, segmented-audio convention, W-level standard retains segmentation, infix `-x-` format, clitic `=`, etc.). Python tests using A's pytest infrastructure.

*Phase B4 — Threshold calibration for soft checks.*
9. The orthography/vocabulary checks emit numbers (Jaccard similarity, overlap coefficient, cosine similarity, Euclidean distance, KL divergence). Today these are presented raw. We need to define, per metric: what range = "all good", what range = "drift worth a comment", what range = "you've probably got the wrong language, dialect, or orthography."
10. Calibration method: compute each metric's distribution across all published corpora believed to be correctly tagged. Look for natural breaks. Sanity-check by deliberately mis-labeling one corpus (e.g., relabel an Amis corpus as Bunun) and confirming the wrong-label case lands in the "probably wrong" range.
11. Bake calibrated thresholds into the validators as defaults, so PR comments are meaningful ("orthography similarity to reference is 0.62, expected ≥0.85 for a correctly-tagged corpus — possible language/dialect/orthography mismatch") rather than raw numbers.

*Phase B5 — Cleaner extensions (per [2026-05-29-clean-xml-extension-tests-design.md](2026-05-29-clean-xml-extension-tests-design.md)).*

The clean_xml extension tests written in A are mostly `xfail` against current behavior. They flip to passing as B implements the following six pieces of infrastructure:

12. **Language-aware cleaning.** `clean_text` and `clean_trans` (or their successor) must consult `xml:lang` on the parent or sibling and branch behavior accordingly. Required: a helper that resolves the "effective" `xml:lang` for any FORM or TRANSL element (walk up to the nearest ancestor with `xml:lang`, fall back to the root `<TEXT xml:lang>`). Affects C001 (FORM/non-Chinese TRANSL only) and C002 (Branch A vs Branch B Chinese-specific behavior).
13. **Warning + CSV output infrastructure.** When a rule emits a warning rather than a transformation, the cleaner appends a row to a per-rule CSV (or one consolidated CSV with rule-ID column). Used by C002 Branch B (single-quote / Latin-quote warnings in Chinese TRANSL), C002b (U+02C8 IPA stress), C012 (segmentation hyphens in orthographies that include `-`), C022 (sentence-initial `*`).
14. **Canonical-orthography TSV lookup at clean time.** C012's data-driven hyphen rule requires the cleaner to read `Orthographies/<canonical>/<Language>.tsv` for the file's `xml:lang` and check whether `-` appears as a letter row. The lookup path (currently `Orthographies/Ortho113/`) should be configurable so switching the canonical orthography only requires a config change. Verified empirically (2026-05-29): the languages where `-` appears as a letter in Ortho113 are **Bunun (`bnn`)** and **Thao (`ssf`)**.
15. **`--hard-remove-segmentation` CLI flag.** Overrides C012's "preserve and warn" branch for orthographies that include `-`, forcing stripping regardless. Default off.
16. **Transformation counter output.** The cleaner keeps counts of every character it transformed (including deletions) and emits a table to the terminal at the end of each run (suggested: rows of `input → output → count`, sorted by count descending). Lets the operator notice unexpected changes.
17. **Existing-corpora remediation (one-off).** Some existing corpora may have been incorrectly stripped of language-appropriate Chinese punctuation by the unconditional `swap_punctuation`. After B implements language-awareness, audit `Corpora/**` for Chinese TRANSL elements that look like they were previously Latin-collapsed, and rewrite them. Companion to the V035 ISO 639-3 remediation in B1 step 6.

18. **Delete `remove_nonlatin` from `clean_xml.py`.** Per user direction (2026-05-29): dead code originally for Wikipedia corpora, since incorporated into the Wikipedia-dedicated pipeline. Simple deletion.

19. **Audit `clean_xml.py` for validator-territory checks.** Per user direction (2026-05-29): if any checks live in the cleaner that belong to validators (gloss-count vs morpheme-count, ISO-639-3 attribute checks, DTD conformance, etc.), move them to the appropriate `validate_*.py` and remove from `clean_xml.py`. Test in the validator suite.

20. **OQ9 docstring update.** Per user direction (2026-05-29): after the language-aware C001/C002 rules land, add a docstring to `clean_text`/`clean_trans` (or successors) documenting the FORM-vs-TRANSL asymmetry.

21. **Dead-branch cleanup.** Surfaced during Task 6d (2026-05-30): `clean_xml.py` lines 198–200 (`if not form_text: root.remove(sentence)`) is unreachable because of a preceding `continue` that fires on empty `form_text`. Delete the dead branch.

22. **Double-walk cleanup.** Surfaced during Task 6d: `clean_xml.py` processes each file twice per invocation — once via the `XML/` subdir walk and once via the root walk. Benign for idempotency but produces duplicated stdout lines. Restructure to walk each file exactly once.

23. **C006 caret-preservation pin revisit.** Surfaced in the 2026-05-30 review of sub-project A. `tests/cleaners/test_clean_xml_extensions.py` pins that the U+2303 caret variant survives in TRANSL today. When B implements C001/C002 language-aware TRANSL rewriting, the C006 pin may need updating to reflect whether ⌃ gets caught by the language-aware path. Not an xfail (the current behavior is "correct" today), but the pin is brittle against B5 work. Revisit the C006 test as part of C001/C002 implementation; either confirm ⌃ continues to survive in TRANSL or update the pin and the design doc together.

*Phase B6 — Category 6 candidates from clean_xml extension transcripts.*

The clean_xml extension design's Category 6 lists patterns Joshua flagged during student-XML reviews that aren't the cleaner's job but ARE real corpus quality issues. Per user direction (2026-05-29), several should be considered for automation somewhere — validators, scrapers, or new tooling. Survey these and decide for each whether B will own it, whether it goes to a scraper pipeline, or whether it stays purely manual review.

21. **C015 — Footnote numbers stuck onto morphemes** (e.g. `uwal12` from a footnote leak). Scraper-side fix per Hunter's Truku card, but a validator that flags suspicious digit-runs adjacent to non-digit text could catch leaks that slip past the scraper.
22. **C016 — Out-of-language examples in a corpus** (Japanese/Fongbe/Tagalog in an Amis or Bunun file). Detection candidate: per-S language identification cross-checked against `TEXT/@xml:lang`. Hard problem; likely WARN-level if attempted.
23. **C021 — Multi-word glosses that should be joined with `.`** (`go to` → `go.to`). Detection candidate: validator that flags space-containing gloss text in TRANSL or M-level FORM. Easy if scoped to morpheme-level glosses.
24. **C023 — `/` indicates alternative forms.** Detection candidate: validator that flags `/` in S-level FORM and recommends scraper-side splitting. The original instruction was scraper-side.
25. **C024 — Parentheses in free translations.** Joshua explicitly said this is too idiosyncratic to generalise. Keep as manual review; do NOT automate.

*Phase B7 — Orthography extraction pipeline (escape handling).*

Per user direction (2026-05-29) surfaced during clean_xml extension design OQ3: the orthography extraction pipeline (`QC/orthography/orthography_extract.py` and consumers) MUST handle XML escape sequences (`&amp;`, `&lt;`, `&gt;`, `&nbsp;`, etc.) by unescaping them before character counting. Without this, each part of an escape sequence is treated as a separate character, polluting orthography statistics and downstream similarity metrics (which feed B4 threshold calibration). Out of scope for clean_xml.py — captured here so it lands when B's orthography work is planned.

**Claude Code support.** A `skill` "validate corpus X" that runs the right sequence with the right flags. A `skill` "compare original vs standard across corpus Y" for B1 discovery. Possibly a `subagent` "spec-vs-DTD reconciler" that systematically walks every rule in the prose spec and reports DTD coverage gaps. A `hook` running fast checks pre-commit is probably overkill — worth a separate decision.

**Dependencies.** **A** (need pytest infrastructure for Phase B3). Phase B1 (discovery) can start before A is done and produces an inventory that informs both A and B3.

**Rough sizing.** B1 discovery: 1–2 sessions plus continuous as new patterns emerge. B2: 1 design + 1 implementation. B3: incremental, grows with the test suite. B4: 1 design + 1 implementation, plus periodic recalibration.

**Open questions.**
- How to handle legacy corpora that fail current validation? Grandfathered with a deny-list? Marked as `xfail` with TODO? Fixed before gating goes live?
- Should `clean_xml.py`/`clean_nonlatin.py` (in-place mutators) be gated behind something stronger than "diff before committing"?
- When the prose spec and the DTD conflict, which is authoritative? Should we always update both together going forward?
- For threshold calibration (B4): is there a "reference corpus" you trust as a calibration baseline, or should we use all published corpora and look for outliers?
- For B1 step 2 (original vs standard comparison): which corpora are most trusted (and so least likely to surface bugs vs. genuine invariants)? Conversely, which are most suspect — those should be looked at *after* the trusted ones, so we have a baseline of what "correct" transformations look like before we encounter sketchy ones.
